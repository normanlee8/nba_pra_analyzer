The following updates were implemented across prop_analyzer/features/generator.py and prop_analyzer/models/training.py to eliminate target leakage, a critical flaw where data from the future or the current outcome was inadvertently visible to the model during training. In generator.py, a .shift(1) operation was added to all player-specific historical aggregations—including rolling averages, median windows, and split averages—ensuring that a player’s statistics for a given game are calculated using only data available prior to that game. Simultaneously, training.py was updated to include a strict feature blacklist and a more refined selection logic that prevents raw, unaggregated box score metrics from the current game (such as TS_PCT or USG_PROXY) from being used as model inputs. These changes solve the leakage problem by mathematically isolating the target variable from the features, which prevents the model from "cheating" by seeing the answer in the data, thereby forcing it to learn genuine predictive patterns and resulting in realistic, reliable performance metrics.
